# Breck configuration library — secrets management, environment setup, and local development

Wed, 20 May 26

### Configuration Management Strategy

- Moving from .env files to Breck configuration system
  - Current problem: secrets scattered across uncommitted .env files
  - Developer drift between local/dev/prod environments
  - No version control for configuration changes
- Breck benefits:
  - Full TypeScript support with autocomplete
  - Configuration errors become compile-time errors
  - All config changes tracked in PRs
  - No more manual .env file sharing between developers

### Security & Key Rotation Implementation

- All secrets moving to AWS Secrets Manager
  - Database connections rotate every 30 days minimum
  - Following SOC 2 requirements for key rotation
  - Lambda functions handle automatic rotation
- Current setup eliminates personal access tokens
  - New developers only need: npm i, npm run dev
  - Requires AWS access but no manual token configuration
  - No CLI tokens or .env files needed

### Breck Configuration Architecture

- File structure hierarchy:
  1. Default configuration (base settings)
  2. Environment-specific (dev/prod folders)
  3. User overrides (personal settings, auto-detected by username)
  4. CLI overrides (runtime parameters)
- Generated config.json file:
  - Built automatically on npm install
  - Contains resolved secrets for runtime
  - Git-ignored build artifact
- Loader system for dynamic values:
  - AWS secrets loader with key/region parameters
  - Supports async operations with blocking resolution
  - Custom loaders possible for other secret sources

### Implementation Plan

- Remove load-config directive from current setup
  - Causes CI/CD to require AWS secrets access
  - Switch to lazy loading on first request
  - Eliminates cold start issues for normal services
- Port Breck to Go for backend services
  - Maintain same configuration hierarchy
  - Enable consistent config management across stack
- Next steps: Start migrating existing services to Breck configuration
